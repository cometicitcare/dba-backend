================================================================================
OBJECTION SYSTEM API - FRONTEND DEVELOPER GUIDE
================================================================================

Base URL: /api/v1/objections

Authentication: Required (Cookie-based session)
Permissions: 
  - objection:create - Create new objections
  - objection:read - View objection details
  - objection:update - Approve/reject/cancel objections

================================================================================
1. CREATE OBJECTION
================================================================================

Endpoint: POST /api/v1/objections/manage
Permission: objection:create

Description:
Create a new objection to restrict operations on an entity. Supports 6 entity 
types and 2 objection types.

Entity Types:
  ‚Ä¢ Vihara (TRN prefix: TRN*)
  ‚Ä¢ Arama (TRN prefix: ARN*)
  ‚Ä¢ Devala (TRN prefix: DVL*)
  ‚Ä¢ Bhikku (REGN prefix: BH*)
  ‚Ä¢ Silmatha (REGN prefix: SIL*)
  ‚Ä¢ High Bhikku (REGN prefix: DBH* or UPS*)

Objection Types:
  ‚Ä¢ REPRINT_RESTRICTION - Block ID card reprint requests
  ‚Ä¢ RESIDENCY_RESTRICTION - Block adding more resident bhikkhus/silmathas

--------------------------------------------------------------------------------
REQUEST PAYLOAD:
--------------------------------------------------------------------------------

Option 1: Using "id" field (Auto-detects entity type from prefix)
{
  "action": "CREATE",
  "payload": {
    "data": {
      "id": "TRN0001234",                           // REQUIRED: Entity ID (TRN/REGN)
      "ot_code": "RESIDENCY_RESTRICTION",           // REQUIRED: Objection type code
      "obj_reason": "Capacity full - cannot add more residents",  // REQUIRED: Reason (1-1000 chars)
      
      // OPTIONAL FIELDS:
      "form_id": "FORM-2025-001",                   // Related form ID/number
      "obj_requester_name": "Temple Admin",         // Requester name (max 200 chars)
      "obj_requester_contact": "0771234567",        // Contact number (max 20 chars)
      "obj_requester_id_num": "199012345678",       // NIC/Passport (max 20 chars)
      "obj_valid_from": "2025-12-01T00:00:00",      // Start date (ISO 8601)
      "obj_valid_until": "2026-12-01T00:00:00"      // End date (null = indefinite)
    }
  }
}

Option 2: Using Direct Entity Identifiers
{
  "action": "CREATE",
  "payload": {
    "data": {
      "vh_trn": "TRN0001234",                       // Vihara TRN
      // OR "ar_trn": "ARN0005678",                 // Arama TRN

2. APPROVE OBJECTION
================================================================================

Endpoint: POST /api/v1/objections/manage
Permission: objection:update

Description:
Approve a PENDING objection. This activates the restriction and BLOCKS the 
relevant operations (reprint requests or adding residents).

Status Transition: PENDING ‚Üí APPROVED
Effect: Restriction becomes ACTIVE (operations blocked)

--------------------------------------------------------------------------------

{

Permission: Any authenticated user

Description:
Check if an entity has an ACTIVE (APPROVED) objection. Use this before showing
"Add Resident" or "Request Reprint" buttons to determine if they should be
disabled.

Entity Type Auto-Detection:
  TRN* ‚Üí Vihara
  ARN* ‚Üí Arama
  DVL* ‚Üí Devala
  BH* ‚Üí Bhikku
  SIL* ‚Üí Silmatha
  DBH*/UPS* ‚Üí High Bhikku

GET /api/v1/objections/check/BH2025000154

FRONTEND INTEGRATION:
--------------------------------------------------------------------------------

JavaScript Example:

async function checkObjectionBeforeAction(entityId) {
  try {
    const response = await fetch(`/api/v1/objections/check/${entityId}`);
    const data = await response.json();
    
    if (data.has_active_objection) {
      // DISABLE action buttons
      document.getElementById('addResidentBtn').disabled = true;
      document.getElementById('requestReprintBtn').disabled = true;
      
      // SHOW warning message
      const objection = data.objection;
      const warningHtml = `
        <div class="alert alert-danger">
          <strong>‚ö†Ô∏è Operation Restricted</strong><br>
          <strong>Type:</strong> ${objection.ot_code}<br>
          <strong>Reason:</strong> ${objection.obj_reason}<br>
          <strong>Approved by:</strong> ${objection.obj_approved_by}<br>
          <strong>Approved at:</strong> ${new Date(objection.obj_approved_at).toLocaleString()}
        </div>
      `;
      document.getElementById('objectionWarning').innerHTML = warningHtml;
    } else {
      // ENABLE action buttons
      document.getElementById('addResidentBtn').disabled = false;
      document.getElementById('requestReprintBtn').disabled = false;
      
      // HIDE warning
      document.getElementById('objectionWarning').innerHTML = '';
    }
  } catch (error) {
    console.error('Error checking objection:', error);
  }
}

// Call before loading entity details page
checkObjectionBeforeAction('TRN0001234');

6. STATUS WORKFLOW
================================================================================

Status Flow Diagram:

PENDING ‚Üí APPROVE ‚Üí APPROVED ‚Üí CANCEL ‚Üí CANCELLED
   ‚Üì                              ‚Üì
REJECT                          CANCEL
   ‚Üì                              ‚Üì
REJECTED                      CANCELLED

Status Descriptions:

‚Ä¢ PENDING (‚è≥ Yellow)
  - Initial status when objection is created
  - Awaiting admin approval/rejection
  - Does NOT block operations yet
  - Can be: APPROVED, REJECTED, or CANCELLED

‚Ä¢ APPROVED (‚ùå Red)
  - Objection is active and blocking operations
  - BLOCKS: Add Resident or Request Reprint (depending on type)
  - Can be: CANCELLED (to remove restriction)

‚Ä¢ REJECTED (‚úÖ Green)
  - Objection was reviewed and denied
  - Does NOT block operations
  - Cannot be changed (final status)

‚Ä¢ CANCELLED (üîµ Blue)
  - Objection was cancelled (from PENDING or APPROVED)
  - If was APPROVED, restriction is now removed
  - Does NOT block operations
  - Cannot be changed (final status)

9. QUICK REFERENCE
================================================================================

Endpoints:
  POST /objections/manage    - All CRUD and workflow operations
  GET  /objections/check/{id} - Check if entity has active objection

Actions (POST /objections/manage):
  CREATE  - Submit new objection (status: PENDING)
  APPROVE - Approve objection (PENDING ‚Üí APPROVED, blocks operations)
  REJECT  - Reject objection (PENDING ‚Üí REJECTED, no restriction)
  CANCEL  - Cancel objection (removes restriction if was APPROVED)

Permissions:
  objection:create - Create new objections
  objection:read   - Read objection details
  objection:update - Approve/reject/cancel objections

Entity ID Prefixes:
  TRN* = Vihara
  ARN* = Arama
  DVL* = Devala
  BH* = Bhikku
  SIL* = Silmatha
  DBH*/UPS* = High Bhikku

Objection Types:
  REPRINT_RESTRICTION - Blocks ID card reprint requests
  RESIDENCY_RESTRICTION - Blocks adding resident bhikkhus/silmathas

Important Notes:
  ‚Ä¢ Only APPROVED objections block operations
  ‚Ä¢ PENDING objections don't block (awaiting approval)
  ‚Ä¢ REJECTED objections don't block (denied)
  ‚Ä¢ CANCELLED objections don't block (restriction removed)
  ‚Ä¢ Cannot create duplicate APPROVED objections for same entity + type
